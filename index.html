Technical Questions
How does Kubernetes handle scaling? Explain HPA vs VPA vs KEDA. ▼
HPA (Horizontal Pod Autoscaler) — Scales replica count based on CPU/memory or custom metrics (Prometheus via KEDA adapter). Best for stateless workloads. Scales out, not up.

VPA (Vertical Pod Autoscaler) — Adjusts resource requests/limits of existing pods. Requires pod restart. Use for workloads where adding replicas doesn't help (e.g., single-threaded processes).

KEDA (Kubernetes Event-Driven Autoscaling) — Scale on external event sources: queue depth (SQS, RabbitMQ), Prometheus metrics, HTTP request rate. Critical for ride-hailing surge patterns — scale driver-matching services when request queue depth exceeds threshold.

For Jeeny specifically: I'd pair KEDA with SQS queue metrics for trip dispatch services — scale to zero during off-peak (2–5AM), burst to 50+ pods at peak (evening rush). Cluster Autoscaler handles node provisioning behind it.

What's your approach to Kubernetes multi-tenancy and namespace isolation? ▼
Logical isolation (namespace-per-team):
— RBAC: each team gets a ClusterRole scoped to their namespace only
— Network Policies: deny-all default, allow only necessary pod-to-pod paths
— Resource Quotas + LimitRanges per namespace to prevent noisy-neighbor
— Separate service accounts with IRSA for cloud resource access

Hard isolation (when compliance demands it):
— Virtual clusters via vCluster, or separate node pools with taints/tolerations
— Separate AWS accounts per environment (Control Tower landing zones)

At ServeLLM: I implemented organization-level data isolation at the application layer (PostgreSQL row-level policies + RBAC middleware) which is the analogue for SaaS multi-tenancy.
How would you set up observability for a ride-hailing platform from scratch? ▼
Three pillars approach:

Metrics (Prometheus + Grafana): Scrape kube-state-metrics, node-exporter, and app /metrics endpoints. Define SLOs as recording rules. Alert via Alertmanager → PagerDuty/Slack. Key metrics for ride-hailing: trip dispatch latency p99, driver availability %, payment success rate.

Logs (ELK or Loki): Fluent Bit DaemonSet collects all pod logs → Elasticsearch (or Loki for cost) → Kibana/Grafana dashboards. Structure logs as JSON with trace IDs from day one.

Traces (OpenTelemetry): Instrument services with OTel SDK, export to Jaeger or Tempo. Critical for distributed tracing across booking → matching → payment → notification chain.

Link all three via trace IDs in logs and metrics labels. Create runbooks for top-5 alert categories. This is exactly the stack I built at TechPrysm for AI workloads, adapted to ride-hailing business metrics.
